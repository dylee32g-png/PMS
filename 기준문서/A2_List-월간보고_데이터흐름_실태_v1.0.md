# A-2. 프로젝트 List ↔ 월간보고 데이터 흐름 실태 v1.0
- 작성: 2026-06-11 / 로드맵 A-2 산출물 (코드 전수 추적 결과)

## 1. 결론 한 줄
두 화면은 **완전히 분리된 장부 2권**을 쓰며, 자동 동기화 장치는 **0개**.
연결 끈은 ① 실행번호(수동 매칭) ② 주간 장부(progressRecords) 공유, 둘뿐.

## 2. 장부 위치
| 화면 | 저장소(Firestore) | 구조 |
|------|------------------|------|
| 프로젝트 List | `projectListRows_{팀}` (행별 문서) + `projectListMeta/{팀}` (헤더) | 엑셀 컬럼명 그대로(한글 키), `_changeHistory`·`_statusHistory` 내장 |
| 월간보고 | `projects` (행별 문서, team 필드로 팀 구분) | 고정 필드(execNo, project, progressStatus, monthlyData[] …) |
| 진행실적(주간) | `progressRecords_{팀}` | **양쪽이 공유** — docKey = 실행번호 ‖ 행ID |

## 3. 중복 기록되는 정보 (양쪽에 따로 적힘 — 동기화 없음)
| 정보 | List 컬럼(동적) | projects 필드 | 비고 |
|------|----------------|---------------|------|
| 프로젝트명 | 프로젝트명/Project/공사명… | project | 실행번호 매칭 시 유사도 비교에 사용 |
| 상태 | 진행현황 (진행중·추진중·완료·취소·삭제·Hold·이전) | progressStatus + monthlyData[].progressStatus (진행·금월완료·신규·미작업·예상·검토중·보고완료…) | **어휘 체계 자체가 다름** → 기준문서 '상태 2단계'와 자연 맵핑(List≈계약현황, 월간≈작업현황) |
| 담당자 | 담당자 (ASSIGNEE_LIST 하드코딩+정규화) | manager | List 정규화 목록은 기술2팀 인명 고정 → 4팀 확대 시 수정 필요 |
| 발주처 | 발주처 | client | |
| 날짜 | 공사계약/공사완료 | startDate/endDate | 의미 주의: 계약일≠시작일일 수 있음 |
| 실행번호 | 실행번호 | execNo | 유일한 연결 키 (수동 등록) |

## 4. 흐름 실태 (코드 확정)
- List → projects **쓰기 0건** (실행번호 등록도 projects를 *읽기만* 하고 List 장부에 기록)
- projects → List **쓰기 0건** (App.js에 projectListRows 접근 코드 없음)
- 화면 점프: 실행번호를 들고 상대 화면 이동+하이라이트 (양방향, 데이터 이동 아님)
- 진행실적 팝업: 양쪽에서 열림, 같은 `progressRecords_{팀}`에 기록

## 5. 어긋남 지점 (함정 4개)
| # | 함정 | 시나리오 |
|---|------|---------|
| T1 | **주간 장부 갈라짐** | 실행번호 미등록 행: List에서 열면 docKey=List행 `_id`, 월간에서 열면 docKey=projects `id` → 같은 프로젝트의 주간 기록이 서로 다른 문서 2개에 쌓임 |
| T2 | **고아 데이터** | 행ID docKey로 키인하다가 나중에 실행번호 등록 → docKey가 바뀌어 이전 기록이 안 보임(문서는 남아있으나 연결 끊김) |
| T3 | **List 팝업엔 '월간 적용' 없음** | List에서 연 진행실적 팝업은 onApplyToMonthly 미전달 → 버튼 자체가 없음(1220행 조건). 주간 장부까지만 적힘. ※ 통일 계산기(6/11) 덕에 월간 화면에서 B폴백으로 보이긴 함 |
| T4 | **메모리 미갱신** | 월간 화면의 progressRecordsMap은 1회성 로드(getDocs, 구독 아님) → List에서 키인해도 월간 화면 재진입 전까지 옛 숫자 |
| T5 | 중복 5종 수동 이중 관리 | 상태·이름·담당자·발주처·날짜를 양쪽에 따로 키인 → 어긋나도 아무도 모름 |

## 6. A-3/A-4로 넘기는 결정 사항
1. **연결 키**: 실행번호(변경 가능·중복 가능·수동)를 유지할지, 내부 고유 ID를 발급해 양쪽 장부+주간 장부 docKey를 통일할지 (Claude 권고: 고유 ID. 실행번호는 '표시용 속성'으로 강등)
2. **어미 확정**: 중복 5종의 원천을 List로 일원화하고 월간보고는 참조(또는 동기화)할지
3. T1~T2 기존 갈라진/고아 데이터의 **정리(마이그레이션)** 범위
