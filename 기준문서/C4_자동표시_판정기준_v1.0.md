- 근거: 자료기준_v1.0 §1-C("신규·금월완료는 상태 아님 → 자동 표시"), §2-B(자동 계산 항목).

## 0. 현장 확인 결과 (2026-06-22, 팀장님)
- '신규' 기준 = **"이번 달 새로 시작한 건"** → 시점 기반 자동판정과 일치.
- '완료' 기록 방식 = **"작업현황을 '완료'로 변경"** → 작업현황 전환으로 금월완료 판정 가능.

## 1. 핵심 발견 (왜 이 설계인가)
- 프로젝트에는 **"처음 들어온 시점"이 저장돼 있지 않음**.
  - 고유 ID(pid)는 2026-06-11 **일괄 발급** → 발급 시각 = 진짜 등록 시점 아님.
  - `createdAt`은 **사용자 계정용**(App.js 685·1849), 프로젝트용 아님.
- 따라서 '신규' 판정엔 **등록 시점을 새로 기록**해야 함. ('최초 등록월' 도입)

## 2. 신규 판정 규칙
- **판정**: `프로젝트의 최초 등록월 == 현재 기준월` → 자동 '신규' 표시.
- **데이터**: `_firstMonth` 필드 신설 (값 예: `'2026-06'`).
  - 신규 등록(handleSubmit의 새 프로젝트 경로)에서 `_firstMonth = 현재 기준월` 저장.
  - **과거 프로젝트**(필드 없음): 읽을 때 `monthlyData`의 **가장 빠른 date**로 추정(on-the-fly). 데이터를 일괄로 건드리지 않아 안전.
  - 판정 함수(예시): `regMonth = p._firstMonth || min(monthlyData.date)` → `regMonth === currMonthStr`.
- **이월 충돌 없음**: 최초 등록월은 한 번 정해지면 불변 → 이월(A안)이 지난달 기록을 복사해도 영향 없음.

## 3. 금월완료 판정 규칙
- **판정**: `이번 달 작업현황 ∈ {완료, 보고완료}` AND `직전 달까지는 완료 아님` → 자동 '금월완료' 표시.
  - "이번 달에 완료로 **전환**된 것"만 잡음(이미 지난달에 끝난 건 제외).
- **데이터**: 추가 필드 불필요. `monthlyData`의 작업현황(workStatus, 없으면 progressStatus→mapLegacyStatus 파생)으로 판정.
- **이월 충돌 처리**: 완료 프로젝트를 다음 달 또 키인해 지난달 '완료'가 이월돼도, "직전 달도 완료" 조건에서 걸러져 금월완료로 **중복 표시 안 됨**.

## 4. 표시 방식
- 신규·금월완료는 **상태가 아니라 자동 마커** → 프로젝트명 셀의 작은 배지로 표시(기존 '임시'·'주간보고 열람중' 배지와 같은 자리, App.js 4256~4257 근처).
- 색: **신규=파랑, 금월완료=초록** (기존 STATUS_COLORS 그대로 — 화면 일관성).
- 진행현황 칸은 3차에서 만든 [상태배지 + 계약·작업현황 보조줄] 그대로 유지.

## 5. 기존 상태값 '신규'·'금월완료' 처리 (안전 전환)
1. **1단계**: 자동 마커를 **추가**(기존 드롭다운 값은 그대로 둠) → 실제 데이터로 자동판정이 맞는지 검증.
2. **2단계**: 검증 통과 후 상태 드롭다운에서 '신규'·'금월완료' **제거**(완전 자동).
- 근거: 곧장 제거하면 자동판정 오류 시 되돌릴 수 없음. 병행으로 안전망 확보.

## 6. 구현 계획 (한 단계씩)
- **4-1** 신규: `_firstMonth` 저장(신규 등록) + 판정 함수 + 프로젝트명 배지.
- **4-2** 금월완료: 판정 함수(작업현황 전환) + 프로젝트명 배지.
- **4-3** 검증 통과 후: 드롭다운에서 두 값 제거(2단계).
- (하위/sub 구조마커는 6/18 '하위' 라벨 작업으로 대부분 완료 — 현황 점검만.)

## 7. 검증 계획 (루틴 규칙8)
- ① 자동 시뮬: 신규(등록월=이번달/아님), 금월완료(이번달 전환/지난달부터 완료/이월복사) 케이스.
- ② 실기기: 새 프로젝트 등록→'신규' 자동 / 작업현황 '완료' 변경→'금월완료' 자동.
- ③ 손검산: 특정 프로젝트 월별기록 vs 화면 마커 대조.
