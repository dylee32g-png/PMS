## 1. 상태값 (2단계 분리) — ★ 구조 확정 (2026-06-10), 세부 용어 검토 중

프로젝트 상태를 한 칸이 아니라 두 칸으로 나눈다.
근거: 기술1팀 양식이 이미 분리 운영(계약현황/작업현황), 기술2팀도 사실상 '추진중'(영업)과 '진행중'(실행)으로 구분해 쓰고 있음.

### 1-A. 계약현황 (일감이 되기까지의 영업 단계)
| 순서 | 상태 | 뜻 |
|------|------|----|
| 1 | 예상 | 비공식 정보 수준, 정식 의뢰 전 (확정 2026-06-10) |
| 2 | 검토 | 의뢰 접수, 검토 중 |
| 3 | 물량 | 물량 산출 중 |
| 4 | 투심 | 투자심의 진행 중 |
| 5 | 견적 | 견적 제출, 결과 대기 |
| 6 | 수주 | 계약 확정 → 작업현황 단계로 |
| - | 보류 | 영업 단계에서 멈춤 |
| - | 취소 | 수주 전 무산 |

### 1-B. 작업현황 (수주 후 실행 단계)
| 순서 | 상태 | 뜻 |
|------|------|----|
| 1 | 대기 | 수주됐으나 작업 전 (미작업) |
| 2 | 진행 | 작업 중 |
| - | 보류 | 실행 중 멈춤 (Hold) |
| 3 | 완료 | 작업 끝 |
| 4 | 보고완료 | 완료 보고까지 끝 — **확정**: 월간보고에 포함됐는지 판단 위해 완료와 구분 유지 |
| - | 취소/삭제 | 실행 중 무산 |

### 1-C. 기존 용어 → 새 기준 매핑 (확인 필요 항목 포함)
| 기존 용어 (출처) | 새 기준 |
|------------------|---------|
| 추진중 (웹·기술2팀) | 계약현황 단계 중 하나 (검토~견적) |
| 진행중 (웹·기술2팀) | 작업현황=진행 |
| Hold/HOLD/보류 | 보류 |
| 미작업 (웹) | 작업현황=대기 |
| 검토중 (웹) | 계약현황=검토 |
| DROP (기술1팀) | 취소 |
| 금월완료 (웹) | 상태 아님 → 완료일이 금월이면 자동 표시 |
| 신규 (웹·엑셀) | 상태 아님 → 전월 보고에 없던 프로젝트면 자동 표시 — 확정 |
| sub (웹) | 상태 아님 → 모(母)공사 밑 세부 공종 구조 표시 — 확정 |
| 예상 (웹·엑셀) | 계약현황=예상 (1단계) — 확정 |
| 이전 (웹·기술2팀) | 상태 아님 → 발주처 변경 이력으로 기록 + 배지 표시 — 확정 (2026-06-10) |
| 완료 vs 보고완료 | 구분 유지 확정 — 월간보고 포함 여부 판단 목적 |

## 2. 공통 항목 (4개 팀 필수) — ★ 초안, 검토 중

### 2-B. 자동 계산 항목 (입력 불필요)
| 항목 | 계산 방법 |
|------|-----------|
| 번호 | 자동 부여 |
| 신규 표시 | 전월 보고에 없으면 자동 |
| 금월완료 표시 | 완료일이 금월이면 자동 |

| 잔여 수량 | Point − 누적 |
| 증감 | 금월 − 전월 |
| 월별 추이 (그래프) | 매달 저장 데이터로 자동 생성 |

## 2-1. 완료/보고완료 전환 규칙
- 완료: 작업 끝 (담당자가 변경)
- 보고완료: 해당 프로젝트가 월간보고에 포함되어 보고된 후 (보고 시점에 일괄 전환 가능 — 검토)

## 3. 팀별 선택 항목 — 초안
| 항목 | 주 사용 팀 |
|------|------------|
| 안전관리비 (금액·제출·기안) | 기술2팀 (LGD 파주) |
| 서브원 교육일지·작업일보 제출 | 기술2팀 (LGD 파주) |

### 설계 원칙 (확정 2026-06-10)
- **항목 추가는 최대한 간단·쉽게.** 항목 정의를 코드에 박지 않고 설정으로 관리 → 나중에 새 항목이 생겨도 코딩 없이(또는 최소 수정으로) 추가 가능하게 설계.

## 4. 이력 기록 항목 — 초안
| 기록 내용 | 예시 |
|-----------|------|
| 변경일시 | 2026-06-10 |
| 변경자 | 염경록 |
| 무엇이 | 상태(작업현황) / 발주처 / 일정 등 |

- 입력 방식: 변경 시 팝업으로 사유 한 줄 입력 (간단하게)

## 8. 동기화 설계 방침 (2026-06-10 확정, 구현 전)
**대원칙: "한 데이터, 두 화면" — 프로젝트 List가 어미(기초 베이스) 자료, 월간보고는 같은 데이터의 보고용 창.**
- 진행률·진행실적 키인은 프로젝트 List와 월간보고 **양쪽 어디서든** 가능, 어디서 입력하든 즉시 상호 동기화.
- 실제 운영은 프로젝트 List 쪽 키인이 다수일 것 → 이 흐름 우선 매끄럽게.

### 연결 키 (Claude 추천안 — 팀장님 내부 확인 예정)
- 현재: 실행번호 숫자 유무로 "진행중" 판단·동기화 → 번호 누락/변경 시 끊김. 실행번호는 키로서 의미 없을 수 있음(팀장님 판단 동일).
- **추천: 내부 고유 ID(프로젝트 생성 시 자동 부여되는 문서 ID)를 유일한 연결 키로 사용.**
  - List·월간보고·진행실적 기록이 전부 같은 ID 하나를 바라봄 (단일 원본).
  - 실행번호·프로젝트명은 표시/검색용 속성으로 격하 — 바뀌어도 연결 안 끊김.
  - 동기화는 Firebase 실시간 반영(onSnapshot, 이미 코드에 기반 있음) 활용.

### 진행실적 등록 팝업 항목 설정화 (메모)
- 항목(도면입수, I/O Map 입수, 화면작성, 기준정보생성, PLC, ETOS, HMI, 자체시운전, 통합시운전)을 **팀 설정에서 인에이블/디세이블** 가능하게.
- UI 정비 단계에서 팝업 UI도 함께 깔끔하게 수정.

## 9. 작업 방식 기준 (2026-06-10 확정)
- **이 시스템은 회사 전체 업무를 취합·보고하는 핵심 도구. 데이터 신뢰가 최우선** — 신뢰가 무너지면 시스템 존재 의미가 없음.
- 건바이건 수정 금지 → **전수검사 + 계산·데이터 로직 한 곳으로 일원화** 방식으로 진행.
- 코드가 한곳에 뭉쳐 수정이 어렵다고 판단되면 **과감히 파트를 나눠(모듈 분리) 병렬 작업** — 회사에서 디자인/프론트/백엔드/기획 팀을 나누듯, 업무 효율과 유지보수 용이성이 목적.
